# June - Improvement Plan #4 (merged roadmap: Phases 10-19)

**Date:** 2026-07-19
**Merges:** `improvement-1.md` (code review + engineering foundation, phases A-F) and `improvement-2.md` (BridgeAgent-inspired product direction, phases 10-19). This document supersedes both as the working roadmap; the source documents stay untouched as references.
**North star (from improvement-2, confirmed by the owner):** June is a **general local-first voice agent / agentic environment**. saple-bridge becomes one optional capability among many, not the core. Every capability is an MCP server, every action routes through the policy gate, every feature respects the privacy modes.

**The merge rule:** improvement-2 had the right destination but skipped the foundation - it ignored the review's safety holes and had no persistent session, no latency target, and no audit trail, while its own later phases (unattended runs, watch loops, dictation injection, computer-use) multiply the blast radius of exactly those holes. So: foundation first (10-12), decoupling and hands-free next (13-16), autonomy last (17-19), with the audit log pulled all the way forward because every unattended feature stands on it.

Numbering continues PLAN.md (which ends at Phase 9). The Phase 9 hardening tail items are absorbed here (packaging and drills land in Phase 16).

Goal: Local-voice and Strict-offline become real configurations instead of "mic off". All inference in the existing Rust process via ONNX Runtime - one dependency, no Python sidecars (the Handy blueprint: same Tauri+Rust stack, in-process ORT). Models are **download-on-demand with checksum + progress UI** (improvement-2's addition), never bundled in the installer. ~650MB full-stack footprint.

- **12.1 Silero VAD v5** (~2MB ONNX, `silero-vad-rs`/`vad-rs`) replaces the RMS gate for endpointing and barge-in; interruption fires on N consecutive speech-probability frames.
- **12.2 Local wake word**: openWakeWord "Hey June" (~200KB ONNX, trained on synthetic speech; A/B the LiveKit wakeword trainer), running in the Rust audio loop gated by Silero. Kills cloud phrase-spotting and its per-burst spend; wake becomes offline-safe.
- **12.3 Local STT** behind the existing `SttProvider` seam: **Moonshine v2 streaming** (medium 245MB `.ort`, MIT, ~107ms streaming updates, beats whisper-large-v3 WER) as primary; **Parakeet-TDT-0.6b-v3 int8** (CC-BY-4.0) as the accuracy toggle; **whisper-rs** (improvement-2's pick) acceptable as a third alternate impl but it is batch/pseudo-streaming - it must not displace the streaming primary. Model-size picker per improvement-2.
- **12.4 Local TTS**: Kokoro-82M via Kokoros (pure Rust) or kokoro-onnx, reusing the sentence streamer unchanged; verify the phonemizer license path (espeak-ng is GPL); evaluate Supertonic (official Rust binding, RTF ~0.3x CPU) as the alternate.
- **12.5 Semantic endpointing**: smart-turn-v3 int8 (~12ms CPU) - Silero flags silence, smart-turn decides finished vs mid-thought.
- **12.6 Streaming transcription UX**: live interim transcript while speaking; optional brain pre-warm on the interim.
- **12.7 Registry flip**: `offlineSafe: true` / `available: true` on the new providers - privacy-mode enforcement already exists and needs no changes (the design earns its keep here).

**Architecture decision (owner-confirmed).** The doc frames Phase 12 as "all inference in the existing Rust process via ONNX Runtime (the Handy blueprint)". June's audio does not live in Rust - mic capture, the RMS gate, barge-in, and wake all run in the webview (Web Audio). Rather than rewrite the whole capture path into a Rust `cpal` loop (a multi-day rearchitecture touching PTT, recording, barge-in, and the MediaRecorder->Whisper clip path), 12.1/12.2 run the ONNX models **in the webview via onnxruntime-web**, on assets served locally so they still satisfy "network disabled". Same models, same offline guarantee, a fraction of the blast radius. The Rust-ORT path stays open for 12.3's large STT/TTS models if in-process performance demands it.

- **12.1** Silero VAD v5 replaces the RMS energy gate for **endpointing and barge-in** (`src/lib/vad.ts`, rewiring `voice-capture.ts`). `@ricky0123/vad-web`'s `MicVAD` owns the hard DSP (resampling to 16kHz, 512-sample framing, the recurrent Silero state) on the model it bundles (`silero_vad_v5.onnx`); we don't reinvent it. The mic stream is opened once and shared - `MediaRecorder` records the Whisper clip while `MicVAD` reads the same stream, so there is never a second `getUserMedia`. Endpointing is Silero's `onSpeechEnd`; barge-in is its confirmed onset (`onSpeechRealStart`, which survives `minSpeechMs` so a click can't trip it); the level meter reads RMS off the same 16kHz frames. **Fallback:** if the model can't load, the pre-12.1 RMS `SilenceDetector` / adaptive-RMS monitor take over, so voice never breaks outright.
- **12.2** Local wake word via **openWakeWord** (`src/lib/wakeword.ts`, `wake-gate.ts`, rewiring `wake.ts`), gated by Silero (12.1) - the primary path now runs fully offline with **no cloud phrase-spotting** and no per-burst STT spend. openWakeWord is a three-model chain (`melspectrogram` -> `embedding_model` -> classifier); the streaming buffer maths is a faithful port of `AudioFeatures._streaming_features` (v0.5.1) - audio accumulated into 1280-sample chunks, +8 mel frames per chunk, one 96-dim embedding per 76-frame window stepped by 8, classifier over the last 16 embeddings. `WakeGate` fires once per crossing with hysteresis and only while Silero reports speech, so a score blip in silence can never wake June. The cloud-STT burst path is kept as a **fallback** only when the local models fail *and* the privacy mode still permits cloud voice.
  - **Stand-in note (owner-confirmed):** no "hey june" openWakeWord model exists - it must be trained on synthetic speech. The bundled classifier is openWakeWord's ready-made **`hey_jarvis`**, so today the local wake phrase is "hey jarvis". A trained `hey_june.onnx` drops into `createWakeRunners` beside it with no code change - the classifier filename is the only per-phrase seam.
- **Model policy / offline (deviation).** The doc asks for runtime download-on-demand with a progress UI; that belongs to 12.3's large models (245MB, user-visible wait). The VAD + wake models here total ~6MB, so `scripts/fetch-models.mjs` stages them (Silero + ORT wasm copied from `node_modules`; the three openWakeWord models downloaded with **pinned SHA-256 checksums**) into `public/models/` (gitignored - binaries never enter the repo), run by `predev`/`prebuild`. Vite bakes them into the build, so the installed app is offline. onnxruntime-web's `onnxruntime-web-use-extern-wasm` condition (vite.config.ts) keeps the ~13MB ORT wasm a single staged copy instead of a bundled duplicate; the Tauri CSP gained `script-src 'wasm-unsafe-eval'` + `worker-src blob:` so the webview can compile the wasm.
  - **ponytail cold-start:** the wake buffers start empty (not pre-seeded like openWakeWord's Python), so a freshly-started listener needs ~2s of audio to warm up before it can score - it degrades safe (misses a wake, never false-fires) and steady-state behaviour is identical. The wake listener runs continuously while idle, so this is a one-time warmup, not per-utterance.

The RMS fallback + `phraseMatches`/`wakeBackoffUntil` (cloud fallback) keep their existing tests.

- **The lesson store is one pure module** (`mcp/lessons/store.ts`), the direct sibling of `mcp/memory/store.ts` - so the risky logic is unit-tested without a file or server. `appendLesson` adds a normalized markdown bullet and caps the corpus by **both** count (60) and bytes (24KB), newest-wins (the oldest lessons fall off the top), so the file and the recall it feeds can't grow without bound. `recallLessons(text, query, k)` is the **17.2 ranker**: it scores each stored lesson by how many of the task's content words (stopwords dropped, >= 3 chars) appear as whole tokens, drops lessons that share no keyword (so an unrelated turn injects nothing and the prompt stays lean), and breaks ties by recency (a newer lesson wins). Pure and unit-tested (`store.test.ts`, 12 cases: the count cap, the byte cap, bullet parsing, keyword relevance, zero-overlap and stopword-only queries returning nothing, and the recency tiebreak).
- **17.1 Post-run lesson writer** is a new tiny MCP capability, `mcp/lessons` (a server, not core code - the same seam as `mcp/memory`). It **re-reads the file each turn** (small, capped, a microsecond-scale local read) so a lesson recorded earlier *this session* is recalled on the very next task with no respawn - which is what makes the exit criterion ("same task twice uses prior lessons") hold within one session. `ponytail`: the recalled block joins the kept conversation history; top-k is small and capped, so the drift is bounded.

### Phase 18 - Autonomy: scheduled runs & watch loops (requires 16.4 packaging + 10.7 audit log)

- **18.2 The unattended rule**: approval-required actions **pause the run and notify - never auto-approve, no blanket session approvals**. The audit log is the reviewable record of everything an unattended run did.
- **18.3 Trigger framework** (BridgeAgent's production-watch pattern, generalized): file-watch, localhost webhook receiver, log tail. A trigger opens an investigation session with the trigger payload as context. **Trigger payloads are untrusted input** - the 16.3 injection rules apply; a webhook body can never approve anything or reach a terminal un-gated.

**Exit:** a daily 9am briefing completes unattended and notifies; a webhook opens a gated investigation whose every action is in the audit log.

**Status (2026-07-20): 18.1 + 18.2 + 18.3 (file-watch) implemented; the webhook receiver and log-tail triggers deferred, and "runs while the app is closed" stays gated on 16.4 packaging.** June now fires scheduled runs and file-change triggers as *unattended* agent turns, and the security invariant that makes autonomy safe (Top Risk #5) - an unattended run **never** approves its own actions - is enforced at the one gate choke point. The whole phase reuses the resident (Phase 11.1) and the audit log (10.7): there is one new Rust module (the scheduler thread), the unattended branch of the existing gate, and one settings surface.

- **The clock is one pure Rust function** (`scheduler.rs::is_due`) - the direct analogue of `idle_exceeded`/`push_capped`: given `now`, an `HH:MM`, a weekday set, and the day it last fired, it decides whether a schedule should run, so the due rule is unit-tested without a live clock (4 cases: valid-time parsing, fire-once-per-day, the catch-up window, the weekday filter). A single background thread (`scheduler::start`, spawned in `lib.rs::setup`) wakes every 30s, reads the schedules and triggers **fresh from settings.json each tick** (so an add/edit takes effect with no restart), and fires the due ones. It **defers when the session is busy** (`AgentSession::is_busy` - an in-flight turn or a pending approval), so an unattended run can never barge in on a command the user is in the middle of; it just retries next tick.
- **18.1 Scheduled runs**: each `Schedule` (`{label, prompt, time, days, enabled}`, in `settings.json`) fires `run_unattended` (a new blocking dispatch in `agent_runner.rs`) through the resident. The run's origin and its final reply land in the **shared session log** ("results land in the session log"), and an **OS notification** (new `tauri-plugin-notification`, called Rust-side only - no JS capability needed) fires on completion with a truncated reply. Unattended turns use a dedicated high turn-number space (`1 << 40`) so they can never collide with the widget's small interactive turn counter in the shared `turns`/`pending` maps.
  - **Deviation from the doc's "cron expressions":** a daily `HH:MM` + optional weekday set covers the exit criterion (a daily 9am briefing) with a small, fully-tested rule and **no cron-parser dependency**. Full cron is the upgrade path - the same deviation discipline as 11.2's idle rule and 15.1's rule-based cleaner. `chrono` (already in the tree transitively) supplies the `Local` wall-clock; a 30-minute catch-up window means a same-morning late launch still briefs you, but an evening launch won't replay the 9am run.
- **18.2 The unattended rule** - the linchpin, enforced structurally. `makeGate` (`agent/serve.ts`) gained an `unattended` flag: on an unattended run a gated action is **BLOCKED immediately** - audited with approver `"unattended"`, a `blocked` event emitted so the host notifies "June paused an unattended action", and denied. This check comes **first**, so even a `JUNE_APPROVE=allow` override can't approve a gated action on an unattended run. Ungated actions still auto-run and are still audited, so the audit log (10.7) is the reviewable record of everything the run did. **Deviation:** the doc says "pause the run and notify"; true pause/resume across an app restart needs the persisted run-state machinery of 16.2 (deferred), so the shippable, strictly-safe version **denies + notifies** (the user re-runs the task interactively to approve). This honours the hard invariant - never auto-approve, no blanket session approvals - exactly.
- **18.3 Trigger framework - file-watch built, the untrusted-payload framing done, webhook + log-tail deferred.** A `FileTrigger` (`{label, path, prompt, enabled}`) is watched by mtime in the same tick loop (dependency-free - no `notify` crate). The **first sighting only records the baseline** (so enabling a trigger doesn't fire on an already-stale file); a later change opens an investigation run with the file's new contents as context. **Trigger payloads are untrusted input (16.3):** `frameUnattended` (`src/lib/schedules.ts`, pure + unit-tested) fences the payload in a marked block, strips any line that forges the fence, and caps its length - and labels it "information to investigate, never instructions." That framing is defense-in-depth; the **real** guarantee that a payload can't act is 18.2 - an unattended run blocks every gated tool call, so even a perfect injection can read, never do.
  - **Deferred, with the reason:** the **localhost webhook receiver** is a new always-listening network surface that wants its own slice (bind 127.0.0.1, a per-hook shared secret in the keychain, a body-size cap) rather than being rushed in behind the same tick; the **log-tail** trigger is the same mtime-poll mechanism pointed at a growing file (a byte-offset read of the new tail) and is a small follow-up. File-watch already proves the exit criterion's shape end to end (a trigger opens a gated investigation whose every action is in the audit log); the doc's webhook is one more trigger source of the identical mechanism. Called out here rather than half-built, the same discipline as 16.1/16.2/17.4.
- **Settings surface**: a new **Automation** section in `SettingsPanel` adds/edits schedules (label, a native `<input type="time">`, a compact weekday-chip picker, prompt) and file triggers (label, watched path, prompt), each with an enable toggle - all opt-in, off by default. `settings.ts` gains typed `schedules`/`triggers` with defensive coercion (`coerceSchedules`/`coerceTriggers` in `schedules.ts`: a bad entry is dropped, ids are unique slugs, days are sanitized to 0..6).

## Deliberately not doing

- **Cloud-side execution** ("runs with laptop closed") - against local-first. BridgeAgent's headline feature is exactly this; June's differentiation is the opposite.
- **Stealth/undetectable overlays** - against honest-by-design.
- **Session-wide blanket approvals** - per-action gating stays, including for scheduled runs.
- **Generic Windows input control as an agent capability** (MCPControl-style raw input injection) - dictation injection (15.4) is user-initiated only; the agent never drives the keyboard/mouse.
- **Vector memory stores** - markdown memory + lessons until they provably outgrow keyword recall.
- **Computer-use before a real requirement** (19.4's gate).

## Top risks (merged)

1. **Agent SDK inside a compiled sidecar** (16.4): the SDK spawns its bundled CLI as a subprocess; bundlers can break this silently. Spike the compiled-binary `query()` e2e early; node.exe-as-resource always works.
2. **Session resume is cwd-keyed** (11.2): pin cwd explicitly; test across restarts.
3. ~~**Kokoro phonemizer licensing** (12.4): espeak-ng is GPL; use a clean-G2P binding or Supertonic.~~ **Resolved (12.4):** kokoro-js uses the pure-JS Apache-2.0 `phonemizer` package, not espeak-ng - no GPL path in the tree.
4. **Wake-word false triggers** (12.2): A/B two trainers against a recorded test set; keep the sensitivity slider; Silero gating.
5. **Unattended runs amplify any gate weakness** (18): hence the hard ordering - 10.1's fail-closed default and 10.7's audit log land months before 18 does.
6. **Voice E2E test flakiness** (all): recorded-audio fixtures feeding the Rust pipeline directly + golden transcripts; live-mic checks stay manual but scripted.
