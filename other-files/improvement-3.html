Improvement Plan #3 · BridgeMind-inspired

Make June a whole-OS
voice agent.

Five phases distilled from a research pass on bridgemind.ai - chiefly BridgeVoice (a Tauri 2 + Rust dictation app, June's closest analog) and BridgeAgent (an autonomous mission runner). Net-new ideas only; deep voice/memory work stays in Plan #1.

📅 2026-07-19 🔎 3-agent web research 🧩 Grounded on the June source tree 🎯 5 phases · ~5-6 weeks

The one insight

June already owns the hard parts - STT, TTS, wake word, hands-free, model choice, privacy modes. The biggest untapped leverage isn't more voice plumbing, it's where the transcript goes. Today it feeds only SAPLE. BridgeVoice's move: "your words land wherever the cursor blinks, in any app." One output path turns June from an assistant you talk to into a dictation tool for the whole OS.

The plan

ordered by leverage · each phase independently shippable
1

The Dictation Surface

NET-NEW MULTIPLIER ~3-5 days

Turn STT output into a general-purpose OS dictation path, independent of SAPLE. Small diffs, high leverage - the reason to read this plan.

#ChangeDetail
1.1Inject at cursornew inject.rs · agent_runner.rs · VoicePanel.tsx Capture foreground HWND at record-start (never end, or June steals focus). On final transcript: apply dictionary, set clipboard, restore focus, send Ctrl+V via SendInput. Blacklist June's own windows. Full Unicode via clipboard.
1.2Output-mode toggleSettingsPanel.tsx · settings.ts Recording Behavior: saple (today) · inject (auto-paste) · clipboard (manual). Off-by-default keeps current behavior. One enum + one branch.
1.3Custom dictionarynew dictionary.ts · stt.ts Post-transcription find/replace before injection and SAPLE. High value for your vocab Whisper butchers: "SAPLE", command & agent names, phase labels. Plain JSON, word-boundary + case-aware, quick-add from history.
1.4Transcription historynew history.ts · AppWindow.tsx Rolling local log of utterances: re-copy, re-inject, correct, and "add to dictionary". Respects privacy modes (no history when forbidden).
1.5Insert-target guardinject.rs Verify saved HWND is still valid & not June's; on mid-utterance app-switch, fall back to clipboard with a toast rather than mis-pasting. Guards the #1 dictation footgun.
Acceptance

Speaking into a focused Notepad/VS Code/browser lands dictionary-corrected text at the cursor with correct Unicode; clipboard mode pastes nothing; a mid-utterance app-switch degrades to clipboard, never mis-pastes; "SAPLE" transcribes right; history can re-inject.

2

Reliability & Onboarding

~2-4 days

BridgeVoice's changelog is mostly hard-won PTT fixes - steal it as a QA checklist. Aligns with "fix flakiness on sight."

#ChangeDetail
2.1Phantom-release guardvoice-capture.ts · wake.tsIgnore a key-up in the first ~150ms of a PTT press - OS hooks emit spurious releases and end recordings instantly.
2.2Silent-stream failoverstt.rs (capture)Detect a dead/all-silence input; rebuild it and fail over to the next device before a "no audio" error. The most common Windows voice-app failure.
2.3Self-heal input hooksvoice-capture.tsIf the OS reclaims the global hotkey, re-register automatically instead of going silently dead until restart.
2.4One-click hotkey setupSettingsPanel.tsx · wake.tsFirst-run "press your key now to test" with a live mic meter + confirmation + conflict detection. Kills the #1 support issue: "the hotkey doesn't fire."
2.5Wake-listener backoffwake.tsStop retrying after N failures (also Plan #1 A7). A bad key currently means silent API spam, one call per speech burst.
Acceptance

Sub-150ms accidental key-up still records; unplugging the mic mid-session fails over without a dialog; a killed hotkey self-recovers; first-run verifies mic + hotkey before the first real command.

~4-6 days deep work → Plan #1 C/B

4

The Mission Loop

THE PRODUCT BET ~1-2 weeks depends on Plan #1 Phase B

BridgeAgent's defining idea: own missions, not prompts - plan → act → monitor → fix → fold the lesson back. June's leap from command-runner to agent. The primitives already exist in saple-memory (tasks/runs/incidents/lessons).

#ChangeDetail
4.1Decomposecore.ts · new mission.tsA spoken multi-step goal is planned into verifiable sub-tasks via create_task, spoken back for confirmation before work begins.
4.2Act + recordcore.ts · agent_runner.rsSub-tasks run through the existing brain/tool loop; progress via append_run_event. Needs Plan #1 B1's resident process - a per-turn spawn can't hold a mission.
4.3Monitor + fixmission.ts · saple-memoryOn failure, capture an incident and attempt one bounded fix before escalating by voice. Hard-destructive steps still gate to a click.
4.4Fold the lessonrecord_lesson · .june-memoryWrite what worked/failed so the next mission "starts sharper" - BridgeAgent's self-rewriting playbook, via June's lesson store. Not fine-tuning.
4.5Mission boardAppWindow.tsxtodo → in-progress → in-review → done (BridgeMCP's lifecycle), so a long unattended run stays legible.
Acceptance

One spoken goal produces a decomposed task list, executes with per-task run events, reports failures by voice with one bounded fix, and writes a visible lesson; a destructive sub-step still requires a click.

5

Integration Bus & Proactive Voice

~3-5 days · mostly config

Give the mission loop reach and a voice of its own. Don't hand-build connectors - MCP is already the bus (Gmail / Calendar / Drive connectors exist in the environment today).

#ChangeDetail
5.1Voice → MCP routercore.ts · policy.tsGeneric path from a voice intent to any attached MCP tool. No per-service code; rides the MCP seam + Plan #1 A1 fail-closed policy (unknown mutating tools ask first).
5.2Wire connectorsconfigAttach the present Gmail / Calendar / Drive MCP connectors as voice-reachable. send-class tools are gated external effects - tool output must never satisfy an approval.
5.3Proactive speechnew announce.ts · agent_runner.rsEvent-driven, unprompted TTS: a finished mission or an MCP watcher error speaks up ("mission done - PR opened"). Reuses the sentence-chunk streamer; opt-in; respects quiet modes.
Acceptance

"What's on my calendar tomorrow" routes to the Calendar MCP and answers by voice; an email-send prompts for approval and a crafted tool result can't auto-approve it; a mission finishing while away triggers a barge-in-able spoken summary.

Sequencing & risks

Order of play

  1. Phase 1 first, always. Smallest diffs, entirely net-new value, broadens June into a general OS tool. Nothing blocks on it.
  2. Phase 2 keeps voice UX honest and interleaves with 1.
  3. Phase 3 - reconcile 3.2/3.3 with Plan #1 B4 to avoid a second memory system.
  4. Phase 4 is the bet, but gated on Plan #1 Phase B (resident agent). Don't start until B1 lands.
  5. Phase 5 rides 4 and A1's policy - mostly configuration once tools have missions to serve.

Top risks

1

Focus/injection races (1.1). Capture HWND at record-start, blacklist June windows, degrade to clipboard on doubt.

2

Two memory systems (3 vs #1 B4). Decide 3.2 is the UI layer over B4 before either ships.

3

Mission needs #1 B first (4). Gate Phase 4 kickoff on B1 landing.

4

Injection via tool output (5). Structural rule: tool results never satisfy approvals; full-parameter confirm before external effects.

5

Dictionary over-correction (1.3). Word-boundary + case-aware; history makes bad rules visible.
